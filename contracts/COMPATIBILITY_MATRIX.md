# Compatibility Matrix

This matrix defines the initial compatibility target for platform, schema, mobile runtime, and plugin API versions.

The matrix starts small because the project is not implemented yet. It should be updated whenever package shape, runtime behavior, API compatibility, or plugin compatibility changes.

## Initial Matrix

| Platform version | Schema version | Mobile runtime min | Mobile runtime max | Plugin API | Notes |
| --- | --- | --- | --- | --- | --- |
| `0.1.x` | `v1` | `0.1.0` | `0.1.x` | `0` | Foundation and contract development. Not production-ready. |

## Version Meanings

`Platform version` covers backend, admin frontend, deployment package compiler, and operational compatibility.

`Schema version` covers JSON Schemas under `contracts/schemas`.

`Mobile runtime min` is the lowest runtime version allowed to apply a package generated by the platform version.

`Mobile runtime max` is the highest runtime version known to be compatible. Use a bounded range so breaking runtime/package behavior cannot be applied silently.

`Plugin API` is the module/plugin extension contract version. It starts at `0` until the extension API is formalized.

## Compatibility Rules

- Mobile clients must reject packages with unsupported `schema_version`.
- Mobile clients must reject packages below `runtime_min_version`.
- Mobile clients must reject packages above `runtime_max_version` unless the runtime explicitly supports that range.
- Backend package publication must validate target channel compatibility before activating a package.
- Rollback activates a previous compatible package; it does not edit a published package.
- Breaking schema changes require a new schema major version folder, for example `v2`.
- Plugin API changes must be reflected in this matrix before plugin/module loading depends on them.

## Release Channel Expectations

| Channel | Intended use | Compatibility expectation |
| --- | --- | --- |
| `dev` | Local and early development packages. | May use current unreleased contracts. |
| `test` | Automated and manual test packages. | Must validate against committed schemas. |
| `staging` | Release candidate packages. | Must match the published matrix row for the candidate version. |
| `production` | User-facing packages. | Must match a released matrix row and pass package validation. |

## Update Triggers

Update this matrix when:

- A schema major version is added.
- A package field is added, removed, renamed, or changes meaning.
- Mobile runtime package parsing changes.
- Sync protocol compatibility changes.
- Plugin/module manifest compatibility changes.
- Platform release version changes.
- A runtime is declared unsafe and package application must be restricted.

## Current Open Decisions

- Exact semantic versioning process for pre-1.0 releases.
- Exact generated type tooling for Python, TypeScript, and Kotlin.
- Whether mobile package APIs remain REST-only or later add persisted GraphQL for specific reads.
- Public schema registry domain, if one is needed.
